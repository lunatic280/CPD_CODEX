## 멀티 에이전트 작업 프로토콜

이 프로젝트에서 기능 구현 요청을 받으면 다음 역할 기반 흐름으로 작업한다.

### 1. project_manager 에이전트

역할:
- 사용자의 목표 기능을 분석한다.
- 작업 범위를 정의한다.
- 필요한 변경 파일을 예측한다.
- backend, frontend 에이전트에게 작업을 분배한다.
- 관련 없는 파일은 수정하지 않도록 제한한다.

산출물:
- 기능 목표 요약
- 작업 범위
- backend 작업 목록
- frontend 작업 목록
- 예상 위험
- 테스트 계획 초안

### 2. backend 에이전트

역할:
- project_manager에게 받은 backend 작업만 수행한다.
- API, 서비스 로직, DB, 인증, 예외 처리, 로깅, 테스트가 필요한지 판단한다.
- 관련 없는 frontend 파일은 수정하지 않는다.

산출물:
- 수정한 backend 파일
- 핵심 구현 내용
- 주의해야 할 설계 포인트
- 테스트 필요 항목

### 3. frontend 에이전트

역할:
- project_manager에게 받은 frontend 작업만 수행한다.
- 화면, 상태 관리, API 호출, 폼 검증, 에러 표시, 라우팅을 처리한다.
- 관련 없는 backend 파일은 수정하지 않는다.

산출물:
- 수정한 frontend 파일
- 핵심 UI/UX 변경
- API 연동 방식
- 테스트 필요 항목

### 4. tester 에이전트

역할:
- backend와 frontend 작업 결과를 검토한다.
- 가능한 경우 테스트, 빌드, 정적 검사, 실행 검증을 수행한다.
- 실패한 테스트와 원인을 정리한다.
- 수정이 필요한 부분을 메인 Codex에게 보고한다.

산출물:
- 실행한 테스트 명령어
- 성공/실패 결과
- 발견한 버그
- 재검증 방법

### 5. 메인 Codex 최종 정리

모든 agent 결과를 기다린 뒤 최종 응답에는 반드시 다음을 포함한다.

- 변경 파일 목록
- 핵심 설계 요약
- backend 변경 사항
- frontend 변경 사항
- 테스트 결과
- 남은 위험
- 사용자가 직접 확인해야 할 부분

## 작업 제한

- 관련 없는 파일은 수정하지 않는다.
- 임의로 대규모 리팩터링하지 않는다.
- 사용자의 목표와 직접 관련 없는 기능을 추가하지 않는다.
- 비밀키, 토큰, 인증 정보는 코드에 하드코딩하지 않는다.
- 불확실한 부분은 추측이라고 명시한다.
- 테스트를 실행하지 못한 경우, 실행하지 못했다고 명확히 말한다.
