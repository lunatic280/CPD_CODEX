# Codex 슬래시 명령어와 멀티 에이전트 사용법

이 문서는 Codex를 터미널 또는 앱에서 사용할 때 자주 쓰는 슬래시 명령어와, 이 프로젝트에서 멀티 에이전트를 활용하는 방법을 자세히 설명한다.

기준:

- 현재 프로젝트: Flutter + SQLite 게시판 앱
- 프로젝트 지침: `AGENTS.md`, `AGENT.md`
- 프로젝트 에이전트 설정: `.codex/config.toml`
- 공식 문서: Codex CLI 슬래시 명령어, Codex Subagents 문서

## 1. 슬래시 명령어란

슬래시 명령어는 Codex 대화 입력창에서 `/`로 시작하는 제어 명령이다.

일반 요청은 Codex에게 작업을 시키는 문장이다.

```text
로그인 화면 UI를 개선해줘.
```

슬래시 명령어는 Codex 세션 자체를 제어한다.

```text
/status
/model
/diff
```

슬래시 명령어는 코드 수정 요청과 다르게 다음 용도로 사용한다.

- 현재 세션 상태 확인
- 모델 또는 추론 강도 변경
- 권한과 승인 방식 변경
- 변경 diff 확인
- 긴 대화 요약
- 새 대화 시작
- 멀티 에이전트 스레드 확인
- MCP, 앱, 플러그인 상태 확인

## 2. 슬래시 명령어 기본 사용법

### 2.1 명령어 목록 열기

Codex 입력창에서 `/`를 입력하면 사용할 수 있는 명령어 목록이 열린다.

사용 방법:

1. Codex 입력창에 `/`를 입력한다.
2. 원하는 명령어 이름 일부를 입력한다.
3. 목록에서 명령어를 선택하거나 Enter를 누른다.

예시:

```text
/status
```

명령어가 생각나지 않으면 `/`만 입력해서 목록을 확인한다. Codex 버전, 실행 환경, 활성화된 기능에 따라 보이는 명령어가 달라질 수 있다.

### 2.2 실행 중인 작업 뒤에 명령 예약하기

작업이 이미 실행 중일 때도 일부 슬래시 명령어는 입력할 수 있다. 이 경우 현재 작업이 끝난 뒤 실행되도록 예약될 수 있다.

예시:

```text
/diff
```

긴 구현 작업을 시킨 뒤 변경 내용을 바로 보고 싶을 때 유용하다.

### 2.3 일반 요청과 섞어 쓰지 않기

슬래시 명령어는 세션 제어용이므로, 일반 작업 요청과 한 줄에 섞기보다 분리해서 쓰는 것이 안전하다.

권장:

```text
/status
```

그 다음:

```text
현재 설정을 기준으로 게시글 검색 기능을 구현해줘.
```

비권장:

```text
/status 확인하고 게시글 검색 기능도 구현해줘
```

## 3. 자주 쓰는 슬래시 명령어

### 3.1 `/init`

현재 디렉터리 기준으로 Codex가 참고할 프로젝트 지침 파일을 만들 때 사용한다.

이 프로젝트에서는 이미 `AGENTS.md`가 만들어져 있으며, 다음 내용을 담고 있다.

- Flutter 앱 프로젝트라는 배경
- SQLite를 사용해야 한다는 요구사항
- Markdown은 한국어로 작성한다는 규칙
- 사용자 명령을 `docs/COMMAND_LOG.md`에 기록한다는 규칙

사용 예시:

```text
/init 이 프로젝트는 Flutter와 SQLite를 사용하는 게시판 앱이고, 모든 Markdown 문서는 한국어로 작성해줘.
```

주의:

- 이미 `AGENTS.md`가 있는 프로젝트에서는 기존 지침을 덮어쓰지 않도록 조심한다.
- 새로 생성된 지침은 반드시 사람이 읽고 프로젝트 상황에 맞게 정리한다.

### 3.2 `/status`

현재 Codex 세션 상태를 확인한다.

확인할 수 있는 항목:

- 사용 중인 모델
- 승인 정책
- 샌드박스 설정
- 읽기/쓰기 가능한 경로
- 토큰 또는 컨텍스트 사용 상태

사용 예시:

```text
/status
```

이 프로젝트에서 유용한 상황:

- Codex가 어떤 모델로 작업 중인지 확인할 때
- 파일을 수정할 수 있는 폴더가 맞는지 확인할 때
- 긴 작업 전에 현재 권한 상태를 확인할 때

### 3.3 `/model`

현재 세션에서 사용할 모델과 추론 강도를 바꿀 때 사용한다.

사용 예시:

```text
/model
```

이 프로젝트의 권장 사용:

- 단순 문서 수정: 빠른 모델 또는 낮은 추론 강도
- 복잡한 기능 설계: 강한 모델 또는 높은 추론 강도
- 멀티 에이전트 기획: `project_manager`는 높은 추론 강도

현재 프로젝트 설정에서는 다음처럼 에이전트별 모델 의도가 들어 있다.

| 에이전트 | 모델 | 추론 강도 | 용도 |
| --- | --- | --- | --- |
| `project_manager` | `gpt-5.5` | `xhigh` | 기능 기획과 작업 분배 |
| `frontend` | `gpt-5.5` | `medium` | Flutter UI 구현 |
| `backend` | `gpt-5.5` | `medium` | SQLite 데이터 계층 구현 |
| `tester` | `gpt-5.5` | `medium` | 분석과 테스트 검증 |

### 3.4 `/permissions`

Codex가 명령 실행이나 파일 수정 전에 사용자 승인을 얼마나 요구할지 조정한다.

사용 예시:

```text
/permissions
```

사용 기준:

- 처음 프로젝트를 열었을 때: 보수적인 권한 사용
- 파일 읽기 중심 분석: 읽기 위주 권한
- 테스트 반복 실행: 필요한 명령만 승인
- 네트워크, 설치, 삭제, Git push: 신중하게 승인

주의:

- 학생 실습 프로젝트에서는 Codex가 마음대로 큰 변경을 하지 않도록 보수적으로 시작하는 것이 좋다.
- 삭제, 초기화, 강제 push 같은 위험한 작업은 명확히 확인한 뒤 승인한다.

### 3.5 `/diff`

현재 Git 변경 내용을 확인한다.

사용 예시:

```text
/diff
```

이 프로젝트에서 유용한 상황:

- Codex가 수정한 파일만 확인하고 싶을 때
- 관련 없는 파일이 바뀌었는지 확인할 때
- 커밋 전에 변경 내용을 검토할 때

함께 쓰기 좋은 일반 요청:

```text
현재 diff를 보고 관련 없는 변경이 있는지 검토해줘.
아직 파일은 수정하지 마.
```

### 3.6 `/review`

현재 작업 트리를 코드 리뷰 관점으로 점검한다.

사용 예시:

```text
/review
```

검토 관점:

- 버그 가능성
- 기능 회귀
- 누락된 테스트
- 보안 또는 데이터 손실 위험
- Flutter 화면 overflow 위험

이 프로젝트에서는 기능 구현 후 다음처럼 요청하면 좋다.

```text
/review
```

그 다음:

```text
Flutter 게시판 앱 기준으로 기능 회귀와 테스트 누락을 우선 검토해줘.
```

### 3.7 `/mention`

특정 파일을 대화에 명확히 붙여서 Codex가 그 파일을 기준으로 답하게 한다.

사용 예시:

```text
/mention lib/screens/login_screen.dart
```

이후 요청:

```text
이 파일의 로그인 로직은 유지하고 UI만 정리해줘.
```

파일을 직접 지정하면 Codex가 엉뚱한 파일을 찾는 시간을 줄일 수 있다.

### 3.8 `/compact`

긴 대화를 요약해 컨텍스트를 정리한다.

사용 예시:

```text
/compact
```

유용한 상황:

- 여러 기능을 연속으로 구현해서 대화가 길어진 경우
- 이전 작업 요약은 필요하지만 세부 로그는 줄이고 싶은 경우
- 컨텍스트가 길어져 응답이 느려지는 경우

주의:

- 요약 과정에서 일부 세부 맥락이 줄어들 수 있다.
- 중요한 요구사항은 `AGENTS.md`, `AGENT.md`, `docs/COMMAND_LOG.md`처럼 파일에 남겨 두는 것이 안전하다.

### 3.9 `/clear`와 `/new`

새 대화 흐름을 시작할 때 사용한다.

`/clear`:

- 터미널 화면과 현재 대화 흐름을 새로 시작한다.

`/new`:

- 같은 CLI 세션 안에서 새 대화를 시작한다.

사용 예시:

```text
/new
```

주의:

- 진행 중인 작업이 있다면 먼저 결과를 정리한다.
- 파일 수정 후에는 `docs/COMMAND_LOG.md` 기록이 끝났는지 확인한다.

### 3.10 `/resume`과 `/fork`

이전 세션을 이어 가거나 현재 세션을 갈라 새 방향을 실험할 때 사용한다.

`/resume`:

- 저장된 이전 대화를 다시 연다.

`/fork`:

- 현재 대화를 복제해 다른 접근을 실험한다.

사용 예시:

```text
/resume
/fork
```

유용한 상황:

- 이전 작업을 이어서 진행할 때
- 같은 기능을 다른 설계로 검토해 보고 싶을 때

### 3.11 `/side`

현재 주 작업을 크게 방해하지 않고 짧은 보조 질문을 할 때 사용한다.

사용 예시:

```text
/side 이 설계에서 SQLite 마이그레이션 위험이 있는지 검토해줘.
```

유용한 상황:

- 메인 구현 흐름을 유지하면서 짧은 검토를 하고 싶을 때
- 특정 설계 결정만 따로 확인하고 싶을 때

### 3.12 `/agent`

멀티 에이전트 작업 중 다른 에이전트 스레드를 확인하거나 전환할 때 사용한다.

사용 예시:

```text
/agent
```

유용한 상황:

- `frontend` 에이전트가 어떤 파일을 수정 중인지 보고 싶을 때
- `tester` 에이전트 결과를 따로 확인하고 싶을 때
- 오래 걸리는 작업의 하위 에이전트 상태를 확인할 때

### 3.13 `/mcp`

현재 사용할 수 있는 MCP 도구를 확인한다.

사용 예시:

```text
/mcp
/mcp verbose
```

이 프로젝트에서 유용한 MCP 예시:

- Figma 디자인 분석
- Figma 스크린샷 조회
- 외부 도구 연결 상태 확인

### 3.14 `/apps`와 `/plugins`

앱 커넥터와 플러그인을 확인할 때 사용한다.

사용 예시:

```text
/apps
/plugins
```

사용 상황:

- Figma 같은 외부 도구를 사용할 때
- 설치된 플러그인 기능을 확인할 때
- 새 도구가 현재 세션에서 보이는지 확인할 때

### 3.15 `/copy`

가장 최근 완료된 Codex 응답을 클립보드에 복사한다.

사용 예시:

```text
/copy
```

유용한 상황:

- 최종 보고를 다른 문서에 옮길 때
- Codex가 만든 명령어 목록을 따로 보관할 때

### 3.16 `/ps`와 `/stop`

백그라운드 터미널 작업을 확인하거나 중단할 때 사용한다.

사용 예시:

```text
/ps
/stop
```

사용 상황:

- 긴 테스트가 아직 실행 중인지 확인
- 실행 중인 개발 서버 또는 테스트를 중단
- 잘못 실행한 명령을 멈춤

### 3.17 `/sandbox-add-read-dir`

Windows에서 현재 샌드박스가 읽을 수 없는 외부 폴더를 읽기 허용할 때 사용한다.

사용 예시:

```text
/sandbox-add-read-dir C:\Users\user\Downloads
```

주의:

- 필요한 폴더만 허용한다.
- 쓰기 권한이 아니라 읽기 권한 추가 용도이다.

### 3.18 `/quit`과 `/exit`

Codex CLI 세션을 종료한다.

사용 예시:

```text
/quit
/exit
```

종료 전 확인할 것:

- 수정한 파일 확인
- `docs/COMMAND_LOG.md` 기록 확인
- 필요한 테스트 결과 확인
- 커밋 또는 push 필요 여부 확인

## 5. CLI 명령과 슬래시 명령어의 차이

슬래시 명령어는 Codex 대화 안에서 입력한다.

```text
/status
```

CLI 명령은 PowerShell 같은 터미널에서 실행한다.

```powershell
codex.cmd --help
codex.cmd resume
codex.cmd review
```

Windows PowerShell에서 `codex`가 실행 정책 때문에 막히면 `codex.cmd`를 사용할 수 있다.

예시:

```powershell
codex.cmd --help
```

이 프로젝트에서 확인한 내용:

- `codex`는 PowerShell 실행 정책 때문에 `codex.ps1` 실행이 차단될 수 있다.
- `codex.cmd --help`는 정상 실행되어 CLI 하위 명령 목록을 확인할 수 있었다.

## 6. 멀티 에이전트란

멀티 에이전트는 하나의 작업을 여러 역할로 나누어 병렬 또는 순차로 처리하는 방식이다.

예시:

- `project_manager`: 작업 계획 수립
- `backend`: SQLite 저장소와 데이터 모델 구현
- `frontend`: Flutter 화면 구현
- `tester`: 분석, 테스트, 회귀 검토
- 메인 Codex: 결과 통합과 최종 보고

멀티 에이전트는 여러 사람이 협업하듯 작업을 나누는 방식이지만, 최종 책임은 메인 Codex가 가진다.

## 7. 멀티 에이전트를 쓰면 좋은 경우

멀티 에이전트가 적합한 작업:

- 기능이 화면과 데이터 계층을 동시에 건드리는 경우
- 설계, 구현, 테스트를 분리하면 더 명확한 경우
- Figma 디자인 반영처럼 분석과 구현, 검증이 모두 필요한 경우
- 큰 리팩터링 전에 여러 관점의 검토가 필요한 경우
- 코드 리뷰와 구현을 병렬로 진행하고 싶은 경우

멀티 에이전트가 과한 작업:

- 문구 하나 수정
- Markdown 한두 줄 수정
- 단일 파일의 작은 버그 수정
- 명령어 하나 실행
- 이미 원인이 명확한 간단한 테스트 수정

이런 작은 작업은 메인 Codex가 직접 처리하는 것이 낫다.

## 8. 이 프로젝트의 에이전트 역할

### 8.1 `project_manager`

역할:

- 사용자 목표 분석
- 작업 범위 정의
- 수정할 파일 예측
- `backend`, `frontend`, `tester` 작업 분배
- 관련 없는 파일 수정 방지

산출물:

- 기능 목표 요약
- 작업 범위
- 역할별 작업 목록
- 예상 위험
- 테스트 계획

사용 예시:

```text
project_manager 에이전트로 게시글 검색 기능 작업 범위를 먼저 정리해줘.
아직 코드는 수정하지 마.
```

### 8.2 `backend`

역할:

- SQLite 테이블과 마이그레이션 검토
- 모델과 저장소 구현
- 인증 또는 데이터 로직 구현
- 데이터 테스트 작성

수정 범위 예시:

- `lib/models/`
- `lib/data/`
- `test/data/`

주의:

- 이 프로젝트는 서버 없는 앱이므로 사용자가 명시하지 않으면 백엔드 서버를 추가하지 않는다.
- Flutter 앱 내부 SQLite 저장을 기준으로 구현한다.

### 8.3 `frontend`

역할:

- Flutter 화면 구현
- 상태 관리
- 폼 검증
- 라우팅
- 로딩, 빈 상태, 에러 상태 UI

수정 범위 예시:

- `lib/screens/`
- `lib/widgets/`
- `lib/main.dart`
- `test/widgets/`

주의:

- 기존 데이터 구조를 임의로 바꾸지 않는다.
- UI 변경 작업에서는 기존 기능과 테스트 키를 최대한 유지한다.

### 8.4 `tester`

역할:

- `flutter analyze` 실행
- `flutter test` 실행
- 기능 회귀 검토
- asset, import, route, validation 문제 확인
- 실패 원인 정리

산출물:

- 실행한 명령어
- 성공 또는 실패 결과
- 발견한 버그
- 재검증 방법

## 12. 파일 소유권을 나누는 방법

멀티 에이전트에서는 같은 파일을 여러 에이전트가 동시에 고치면 충돌 위험이 커진다.

좋은 분담:

| 에이전트 | 담당 파일 |
| --- | --- |
| `backend` | `lib/data/post_repository.dart`, `lib/models/post.dart`, `test/data/post_repository_test.dart` |
| `frontend` | `lib/screens/home_screen.dart`, `lib/screens/post_form_screen.dart`, `lib/widgets/post_list_item.dart` |
| `tester` | 직접 수정하지 않고 분석과 테스트 중심 |

주의가 필요한 파일:

- `lib/main.dart`: 앱 전체 라우팅과 테마가 있어 충돌 가능성이 높다.
- `pubspec.yaml`: 의존성 변경은 전체 프로젝트에 영향을 준다.
- `docs/COMMAND_LOG.md`: 메인 Codex가 최종적으로 정리하는 것이 좋다.
- `.codex/config.toml`: 에이전트 설정 변경 요청이 있을 때만 수정한다.

## 13. tester 에이전트에게 줄 검증 지시

## 18. 작업 난이도별 추천 사용 방식

Codex를 잘 쓰려면 모든 작업에 같은 방식을 쓰지 않는 것이 중요하다. 작업 크기와 위험도에 따라 메인 Codex, 슬래시 명령어, 멀티 에이전트를 나누어 사용한다.

| 작업 유형 | 추천 방식 | 이유 |
| --- | --- | --- |
| 문구 수정, README 한 줄 수정 | 메인 Codex 직접 처리 | 역할 분담 비용이 더 크다. |
| 단일 화면의 작은 UI 수정 | 메인 Codex 직접 처리 후 `/diff` 확인 | 변경 범위가 작고 검토가 쉽다. |
| `flutter analyze` 오류 수정 | 메인 Codex 직접 처리 | 실패 원인과 수정 범위가 좁은 경우가 많다. |
| 로그인/회원가입 같은 기능 구현 | 멀티 에이전트 | 화면, 저장소, 테스트가 함께 바뀔 수 있다. |
| 게시글 CRUD, 검색, 카테고리 | 멀티 에이전트 | SQLite와 UI 흐름을 나누어 검토해야 한다. |
| Figma 디자인 반영 | frontend 중심 멀티 에이전트 | 디자인 분석, Flutter 구현, 테스트 검증이 분리된다. |
| GitHub 연결, push, reset | 메인 Codex 직접 처리 + 명시적 확인 | 저장소 상태 변경은 위험도가 높다. |
| 대규모 구조 변경 | 먼저 `/plan` 또는 project_manager 분석 | 구현 전에 위험과 범위를 정해야 한다. |

실습에서는 작은 작업은 메인 Codex로 처리하고, 여러 파일과 계층을 건드리는 작업부터 멀티 에이전트를 사용하는 방식이 좋다.

## 19. 승인과 샌드박스 안전 기준

Codex는 파일을 읽고 쓰거나 명령을 실행할 수 있다. 따라서 권한과 승인 범위를 이해해야 한다.

### 19.1 승인 없이 진행해도 비교적 안전한 작업

- `rg`로 파일 검색
- `Get-Content`로 파일 읽기
- `git status --short`
- `git diff`
- Markdown 문서 작성
- Flutter 코드의 좁은 범위 수정

### 19.2 사용자 확인이 필요한 작업

- 새 패키지 설치
- 네트워크 다운로드
- Android SDK, Flutter SDK, Gradle 캐시 변경
- Git commit, push
- 대량 파일 생성
- 데이터베이스 스키마 변경

### 19.3 특히 조심해야 하는 작업

- `git reset --hard`
- `git checkout -- .`
- 파일 또는 폴더 삭제
- 강제 push
- 비밀키, 토큰, 인증 정보 수정
- 사용자 홈 디렉터리나 시스템 폴더 변경

이런 작업은 Codex에게 바로 실행하게 하지 말고 먼저 다음처럼 요청한다.

```text
이 작업이 어떤 파일을 바꾸는지 먼저 설명해줘.
아직 실행하지 마.
```

또는:

```text
삭제나 reset 없이 해결할 수 있는 방법을 먼저 제안해줘.
```

## 20. 좋은 프롬프트 작성 체크리스트

Codex에게 작업을 맡길 때는 목표만 말하는 것보다 제한과 검증 기준을 함께 주는 것이 좋다.

### 20.1 최소 체크리스트

요청 전에 다음 항목을 확인한다.

- 목표가 한 문장으로 명확한가
- 수정해도 되는 파일 범위가 있는가
- 유지해야 하는 기존 기능을 적었는가
- 새 패키지 설치 가능 여부를 적었는가
- 실행해야 할 테스트를 적었는가
- 최종 보고 형식을 적었는가
- `docs/COMMAND_LOG.md` 기록이 필요한 프로젝트라는 점을 고려했는가

### 20.2 좋은 요청 예시

```text
목표: 게시글 검색 기능 구현

유지할 것:
- 기존 게시글 작성, 상세, 수정, 삭제 기능
- SQLite 로컬 저장 방식
- 서버 미사용 구조

수정 제한:
- 관련 없는 디자인 변경은 하지 마.
- 새 패키지는 필요할 때만 제안하고 바로 설치하지 마.

검증:
- C:\flutter\bin\flutter.bat analyze 실행
- C:\flutter\bin\flutter.bat test 실행

마지막 보고:
- 변경 파일
- 핵심 설계
- 테스트 결과
- 남은 위험
```

### 20.3 애매한 요청을 구체화하는 방법

애매한 요청:

```text
앱 예쁘게 해줘.
```

구체적인 요청:

```text
게시글 목록, 상세, 작성/수정 화면의 UI를 Material 3 기준으로 개선해줘.
기존 CRUD 기능과 SQLite 구조는 유지하고,
모바일에서 제목, 작성자, 날짜, 본문 요약이 읽기 쉽게 보여야 해.
```

## 21. 멀티 에이전트 결과 검수 기준

멀티 에이전트 작업은 결과가 여러 곳에서 나오므로 메인 Codex가 반드시 통합 검수해야 한다.

### 21.1 project_manager 결과 검수

확인할 것:

- 목표를 정확히 이해했는가
- 작업 범위가 너무 넓지 않은가
- backend와 frontend 작업이 겹치지 않는가
- 테스트 계획이 실제 실행 가능한가
- 관련 없는 파일 수정 제한이 들어 있는가

### 21.2 backend 결과 검수

확인할 것:

- 서버를 새로 추가하지 않았는가
- SQLite 저장 방식이 유지되는가
- 모델, repository, database 변경이 일관적인가
- 마이그레이션이 필요한 변경인지 판단했는가
- 데이터 테스트가 필요한 경우 추가했는가

### 21.3 frontend 결과 검수

확인할 것:

- 기존 기능을 보존했는가
- 화면 이동이 깨지지 않았는가
- 입력 검증과 에러 표시가 유지되는가
- 모바일 화면에서 overflow 가능성이 없는가
- 새 asset 또는 package를 임의로 추가하지 않았는가

### 21.4 tester 결과 검수

확인할 것:

- 실제로 실행한 명령어가 적혀 있는가
- 성공과 실패가 구분되어 있는가
- 실패 원인이 파일/라인 단위로 설명되는가
- 재검증 방법이 있는가
- 테스트하지 못한 항목을 명확히 말했는가

## 22. 실패 상황별 복구 방법

### 22.1 `flutter analyze`가 실패한 경우

요청 예시:

```text
flutter analyze 실패 로그를 기준으로 원인을 분류해줘.
자동 수정 가능한 것만 먼저 고치고, 설계 판단이 필요한 것은 보고해줘.
```

확인할 것:

- import 누락
- 사용하지 않는 변수
- const 위치 오류
- null safety 오류
- deprecated API 사용

### 22.2 `flutter test`가 실패한 경우

요청 예시:

```text
실패한 테스트 이름과 실패 메시지를 기준으로
기능 버그인지 테스트 기대값 변경인지 구분해줘.
기능이 맞다면 테스트만 수정하고, 기능이 깨졌다면 코드를 수정해줘.
```

확인할 것:

- 위젯 key 변경 여부
- 화면 문구 변경으로 인한 finder 실패
- async 대기 누락
- repository 동작 변경
- SQLite 초기화 문제

### 22.3 에이전트가 관련 없는 파일을 수정한 경우

요청 예시:

```text
현재 diff를 확인해서 이번 작업과 관련 없는 변경을 구분해줘.
관련 없는 변경은 되돌리지 말고 목록만 보고해줘.
```

주의:

- Codex가 만들지 않은 사용자 변경을 함부로 되돌리면 안 된다.
- 되돌리기가 필요하면 파일별로 사용자 확인을 먼저 받는다.

### 22.4 멀티 에이전트 결과가 충돌한 경우

요청 예시:

```text
backend와 frontend 결과가 충돌하는 부분을 정리해줘.
어느 파일에서 충돌하는지, 어떤 설계를 선택해야 하는지 먼저 보고하고 아직 수정하지 마.
```

검토 기준:

- 같은 파일을 동시에 수정했는가
- 데이터 모델 이름이 서로 다른가
- route 또는 callback 계약이 달라졌는가
- 테스트가 어느 구현을 기준으로 작성됐는가

### 22.5 PowerShell에서 `codex` 실행이 막히는 경우

이 프로젝트에서는 `codex --help`가 PowerShell 실행 정책 때문에 실패할 수 있었다. 이때는 `codex.cmd`를 사용한다.

```powershell
codex.cmd --help
```

이 문제는 `codex.ps1` 스크립트 실행 정책과 관련될 수 있다. 수업 중에는 정책 변경보다 `codex.cmd` 사용을 우선 권장한다.

## 23. 학생 실습 과제 예시

### 23.1 초급 과제

목표:

- `AGENTS.md`를 읽고 프로젝트 규칙 요약하기
- `/status`로 현재 세션 상태 확인하기
- `docs/COMMAND_LOG.md`에 기록이 남는지 확인하기

요청 예시:

```text
AGENTS.md와 docs/COMMAND_LOG.md를 읽고 이 프로젝트에서 지켜야 할 규칙을 요약해줘.
아직 파일은 수정하지 마.
```

### 23.2 중급 과제

목표:

- 게시글 목록 화면의 빈 상태 문구 개선
- `flutter analyze` 실행
- 변경 파일과 테스트 결과 보고

요청 예시:

```text
게시글 목록이 비어 있을 때 보이는 문구를 더 친절하게 바꿔줘.
기존 CRUD 기능은 유지하고, flutter analyze를 실행해줘.
```

### 23.3 고급 과제

목표:

- 멀티 에이전트로 게시글 검색 기능 구현
- backend/frontend/tester 역할 분리
- 최종 보고 검토

요청 예시:

```text
목표: 게시글 검색 기능 구현
AGENTS.md의 멀티 에이전트 작업 프로토콜에 따라 진행해.
backend는 SQLite 검색 로직을, frontend는 검색 UI를, tester는 analyze/test를 담당해줘.
관련 없는 파일은 수정하지 마.
```

### 23.4 리뷰 과제

목표:

- `/diff`로 변경 내용 확인
- `/review`로 회귀 위험 점검
- 남은 위험을 문서로 정리

요청 예시:

```text
현재 diff를 보고 이번 작업의 위험과 테스트 누락을 리뷰해줘.
수정하지 말고 findings만 먼저 알려줘.
```
