# Codex 사용 가이드

이 문서는 이 Flutter 게시판 앱 프로젝트에서 Codex를 처음 사용하는 학생들이 기본 튜토리얼부터 응용 작업까지 단계적으로 익힐 수 있도록 정리한 안내서이다.

슬래시 명령어와 멀티 에이전트 사용법을 더 자세히 보려면 [Codex 슬래시 명령어와 멀티 에이전트 사용법](CODEX_SLASH_COMMANDS_AND_MULTI_AGENT.md)을 함께 참고한다.

## 1. Codex가 하는 일

Codex는 프로젝트 파일을 읽고, 코드를 수정하고, 명령어를 실행하고, 테스트 결과를 바탕으로 다음 작업을 이어 갈 수 있는 개발 보조 도구이다.

이 프로젝트에서는 특히 다음 작업에 사용한다.

- Flutter 앱 코드 작성
- SQLite 기반 로컬 데이터 저장 기능 구현
- 로그인, 회원가입, 게시글 CRUD 같은 기능 추가
- UI/UX 개선
- 테스트 실행과 오류 수정
- 작업 내역과 명령 기록 문서화

## 2. 기본 사용 튜토리얼

### 2.1 현재 프로젝트 파악시키기

처음에는 Codex가 프로젝트 구조를 먼저 읽게 하는 것이 좋다.

예시 요청:

```text
현재 Flutter 프로젝트 구조를 파악하고 주요 파일 역할을 설명해줘.
```

좋은 요청 방식:

- 어떤 파일을 먼저 봐야 하는지 지정한다.
- 코드를 수정하지 말고 분석만 하라고 말한다.
- 최종 결과에 변경 파일이 없어야 한다고 명확히 말한다.

예시:

```text
lib 폴더와 test 폴더를 읽고 현재 앱 구조를 설명해줘.
아직 코드는 수정하지 마.
```

### 2.2 작은 코드 수정 요청하기

처음에는 작은 단위로 요청하는 것이 안전하다.

예시:

```text
로그인 화면의 버튼 문구를 "로그인하기"로 변경해줘.
관련 테스트가 있으면 함께 수정하고 flutter analyze를 실행해줘.
```

좋은 요청에는 다음 정보가 들어간다.

- 바꾸고 싶은 화면 또는 기능
- 유지해야 하는 기존 동작
- 수정하면 안 되는 범위
- 실행해야 하는 검증 명령어

### 2.3 수정 결과 확인하기

Codex에게 작업 후 결과를 다음 형식으로 정리하라고 요청하면 검토하기 쉽다.

```text
마지막에 변경 파일, 핵심 변경 내용, 테스트 결과, 남은 위험을 정리해줘.
```

## 3. Flutter 작업 요청 방법

### 3.1 화면 추가 요청

새 화면을 만들 때는 화면의 목적, 이동 경로, 필요한 상태를 함께 적는다.

예시:

```text
게시글 작성 화면을 추가해줘.
제목과 본문 입력칸이 필요하고, 저장하면 목록 화면으로 돌아가야 해.
기존 게시글 저장소 구조를 우선 재사용해줘.
```

### 3.2 기존 화면 수정 요청

기존 화면을 수정할 때는 기능 유지 조건을 명확히 적는다.

예시:

```text
게시글 목록 화면 UI를 더 읽기 쉽게 개선해줘.
기존 글 목록 조회, 상세 이동, 삭제 기능은 유지해.
데이터 모델은 바꾸지 마.
```

### 3.3 Flutter 명령어 실행 요청

이 프로젝트에서는 Flutter가 `C:\flutter\bin\flutter.bat` 경로에 설치되어 있을 수 있다.

예시 요청:

```text
C:\flutter\bin\flutter.bat analyze를 실행하고 오류가 있으면 수정해줘.
```

자주 쓰는 명령어:

```powershell
C:\flutter\bin\flutter.bat pub get
C:\flutter\bin\flutter.bat analyze
C:\flutter\bin\flutter.bat test
C:\flutter\bin\flutter.bat run
```

## 4. SQLite 기능 작업 요청 방법

이 프로젝트는 서버 없이 앱 내부 SQLite에 데이터를 저장한다.

SQLite 관련 작업을 요청할 때는 다음 항목을 구체적으로 적는다.

- 저장할 데이터 이름
- 필요한 컬럼
- 생성, 조회, 수정, 삭제 중 필요한 기능
- 기존 데이터와의 관계
- 테스트로 확인할 조건

예시:

```text
게시글에 좋아요 수를 저장하는 기능을 추가해줘.
SQLite posts 테이블에 like_count 컬럼을 추가하고,
기존 게시글 CRUD 흐름은 유지해.
저장소 테스트도 추가해줘.
```

주의할 점:

- 사용자가 요구하지 않으면 백엔드 서버를 추가하지 않는다.
- DB 스키마 변경은 기존 데이터 마이그레이션 영향을 확인해야 한다.
- 저장소 테스트를 함께 요청하는 것이 좋다.

## 5. 테스트와 디버깅 요청 방법

### 5.1 테스트 실행 요청

가장 기본 검증은 정적 분석과 테스트이다.

예시:

```text
flutter analyze와 flutter test를 실행하고 실패 원인을 수정해줘.
```

### 5.2 오류 메시지 기반 요청

오류를 붙여 넣을 때는 원문을 그대로 주는 것이 좋다.

예시:

```text
아래 오류가 발생했어. 원인을 분석하고 최소 수정으로 해결해줘.

오류:
...
```

### 5.3 기능 회귀 확인 요청

UI만 수정했더라도 기존 기능이 깨질 수 있다.

예시:

```text
로그인 화면 디자인 변경 후 기존 validator, submit, route 이동이 유지됐는지 확인해줘.
```

## 6. 멀티 에이전트 사용 방법

작업 범위가 커지면 역할 기반으로 에이전트를 나누어 요청할 수 있다.

이 프로젝트의 기본 역할은 다음과 같다.

- `project_manager`: 목표 분석, 작업 범위 정의, 역할 분담
- `backend`: 데이터, 저장소, 인증, SQLite 관련 구현
- `frontend`: Flutter 화면, 상태, 입력 검증, 라우팅 구현
- `tester`: 분석, 테스트, 회귀 위험 검토

예시 요청:

```text
목표: 게시글 검색 기능 구현
AGENTS.md의 멀티 에이전트 작업 프로토콜에 따라 진행해.

project_manager는 작업 범위를 정리하고,
backend는 SQLite 검색 로직을 구현하고,
frontend는 검색 UI를 구현하고,
tester는 flutter analyze와 flutter test를 실행해줘.
```

멀티 에이전트를 사용할 때 좋은 조건:

- 역할별 수정 범위를 명확히 한다.
- 관련 없는 파일은 수정하지 말라고 적는다.
- 모든 에이전트 결과를 기다리라고 적는다.
- 마지막 보고 형식을 지정한다.

## 7. Figma 디자인 적용 요청 방법

Figma 디자인을 Flutter 코드에 반영할 때는 디자인 분석과 코드 수정을 분리하면 좋다.

### 7.1 분석만 요청하기

```text
Figma Remote MCP로 이 프레임을 분석해줘.
whoami, get_design_context, get_screenshot, get_variable_defs를 실행하고
아직 코드는 수정하지 마.
```

### 7.2 기존 화면에 디자인 적용하기

```text
기존 로그인 화면의 기능은 유지하고,
Figma 디자인 기준으로 UI만 교체해줘.
Flutter/Dart 코드만 작성하고 새 패키지는 설치하지 마.
```

확인해야 할 항목:

- 기존 `TextEditingController` 유지
- 기존 `Form`과 `validator` 유지
- 기존 submit 함수 유지
- loading, error, route 이동 유지
- 새 asset 또는 package 추가 여부 확인

## 8. Git과 GitHub 작업 요청 방법

Git 작업은 변경 범위 확인 후 진행하는 것이 안전하다.

예시 요청:

```text
현재 변경 파일을 확인하고 커밋 메시지를 추천해줘.
아직 commit이나 push는 하지 마.
```

커밋까지 요청하는 예시:

```text
현재 변경 사항을 확인하고 관련 파일만 커밋해줘.
커밋 메시지는 "로그인 화면 Figma 디자인 반영"으로 해줘.
```

주의할 점:

- 관련 없는 사용자 변경은 되돌리지 않는다.
- 커밋 전 `git status --short`로 변경 파일을 확인한다.
- push는 원격 저장소와 브랜치를 확인한 뒤 요청한다.

## 9. 명령 기록 작성 규칙

이 프로젝트에서는 사용자가 입력한 모든 명령과 요청을 `docs/COMMAND_LOG.md`에 기록한다.

기록할 내용:

- 날짜
- 사용자 명령 또는 요청 원문
- 수행한 작업
- 생성 또는 수정된 파일
- 검증 결과 또는 차단 사항

사용자가 여러 줄로 입력한 요청은 Markdown 표 안에서 `<br>`을 사용해 줄바꿈을 유지한다.

좋은 요청 예시:

```text
이 작업도 docs/COMMAND_LOG.md에 내가 입력한 문장 그대로 기록해줘.
```

## 10. 좋은 요청 템플릿

### 10.1 기능 구현 템플릿

```text
목표: [기능 이름] 구현

요구사항:
- [필수 동작 1]
- [필수 동작 2]

유지할 것:
- [기존 기능]

수정 제한:
- 관련 없는 파일은 수정하지 마.
- 데이터 구조는 필요한 경우에만 바꿔.

검증:
- flutter analyze 실행
- flutter test 실행

마지막 보고:
- 변경 파일
- 핵심 설계
- 테스트 결과
- 남은 위험
```

### 10.2 버그 수정 템플릿

```text
아래 버그를 수정해줘.

현상:
[사용자가 본 문제]

재현 방법:
1. [첫 번째 단계]
2. [두 번째 단계]

기대 동작:
[정상 동작]

제한:
- 최소 수정으로 해결해.
- 관련 없는 리팩터링은 하지 마.
```

### 10.3 문서 작성 템플릿

```text
[주제]에 대한 문서를 docs 폴더에 한국어 Markdown으로 작성해줘.
초보자가 따라 할 수 있게 단계별 예시를 포함해줘.
작업 후 docs/COMMAND_LOG.md에도 기록해줘.
```

## 11. 응용 사용 팁

- 큰 기능은 한 번에 요청하기보다 분석, 구현, 테스트로 나누어 요청한다.
- 디자인 작업은 기존 기능 유지 조건을 먼저 적는다.
- DB 작업은 스키마 변경과 테스트 필요 여부를 함께 묻는다.
- 실패 로그는 요약하지 말고 원문에 가깝게 제공한다.
- Codex가 실행한 테스트가 실제 기기 테스트를 대체하지는 않는다.
- 최종 보고에서 남은 위험을 반드시 확인한다.

## 12. 이 프로젝트에서 추천하는 작업 흐름

1. 요구사항을 짧게 정리한다.
2. Codex에게 관련 파일을 먼저 읽게 한다.
3. 수정 범위와 제한을 명확히 말한다.
4. 작은 단위로 구현한다.
5. `flutter analyze`를 실행한다.
6. `flutter test`를 실행한다.
7. 결과를 `docs/COMMAND_LOG.md`에 기록한다.
8. 실제 Android 기기에서 핵심 흐름을 직접 확인한다.

이 흐름을 따르면 학생들이 Codex가 어떤 판단을 했고, 어떤 파일을 바꿨고, 어떤 검증을 했는지 따라가기 쉽다.
